## Bad practices в Docker Compose

### 1 bp

```yaml
image: python:latest
image: postgres:latest
```

**Почему плохо:** latest меняется со временем, сборка/запуск становятся непредсказуемыми.
**Как была исправлена:** использованы фиксированные версии: `python:3.12-slim`, `postgres:16.4`.
**Как повлияло на результат:** запуск стал воспроизводимым и стабильным.

### 2 bp

```yaml
privileged: true
```

**Почему плохо:** контейнер получает избыточные привилегии, что увеличивает риски безопасности.
**Как была исправлена:** параметр `privileged` удален.
**Как повлияло на результат:** уменьшена поверхность атаки и повышена безопасность запуска.

### 3 bp

```yaml
environment:
  DB_PASSWORD: super_secret_password
  POSTGRES_PASSWORD: admin123
```

**Почему плохо:** секреты хранятся в файле в открытом виде.
**Как была исправлена:** пароль БД передается через переменные окружения:
`POSTGRES_PASSWORD: ${POSTGRES_PASSWORD:?Set POSTGRES_PASSWORD in .env}`.
**Как повлияло на результат:** уменьшен риск утечки чувствительных данных.

### 4 bp

```yaml
ports:
  - '5432:5432'
```

**Почему плохо:** порт базы данных открыт наружу без необходимости.
**Как была исправлена:** публикация порта БД удалена.
**Как повлияло на результат:** меньше лишних внешних точек доступа.

### 5 bp

```yaml
volumes:
  - ./:/app
```

**Почему плохо:** широкий bind-mount копирует в контейнер весь проект, что может приводить к конфликтам и случайной подмене файлов.
**Как была исправлена:** bind-mount удален; для БД используется named volume `db_data`.
**Как повлияло на результат:** данные БД сохраняются между перезапусками, конфигурация стала чище и безопаснее.

## Сетевая изоляция в хорошем compose

**Как сделана изоляция:**
В `docker-compose.good.yml` сервисы запускаются в рамках одного compose-проекта, но подключены к разным сетям:

- `web` подключен только к `web_net`
- `db` подключен только к `db_net`

Обе сети объявлены отдельно и имеют `internal: true`.

**Принцип изоляции:** контейнеры видят только сервисы в общей с ними сети. Так как у `web` и `db` нет общей сети, они не могут обращаться друг к другу по имени сервиса или IP внутри проекта, хотя запускаются вместе одной командой.
