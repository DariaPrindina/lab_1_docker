## Bad practices Dockerfile

### 1 bp

```docker
FROM python:latest
```

**Почему плохо:** latest меняется со временем, сборка сегодня и через неделю может дать разный результат
**Как была исправлена:** зафиксирован тег/версия - `python:3.12-slim`
**Как повлияло на результат:** сборка стала стабильной и предсказуемой

### 2 bp

```docker
ENV APP_ENV=production
ENV DB_PASSWORD=super_secret_password
```

**Описание:** Секреты в Dockerfile
**Почему плохо:** секрет попадает в образ и его историю, может утечь
**Как была исправлена:** не храним секреты в Dockerfile, передаем их при запуске
**Как повлияло на результат:** уменьшился риск утечки чувствительных данных

### 3 bp

```docker
RUN apt-get update
RUN apt-get install -y curl vim
RUN pip install --no-cache-dir flask
```

**Описание:** Установка пакетов в нескольких `RUN` и без очистки apt-кэша
**Почему плохо:** каждый `RUN` создает новый слой, образ раздувается, кэш apt остается внутри и увеличивает размер
**Как была исправлена:** объединены команды в один RUN и очищен кэш
**Как повлияло на результат:** уменьшился размер образа и ускорилась сборка и загрузка

### 4 bp

```docker
COPY . .
```

**Почему плохо:** в образ копируется все из директории проекта, включая лишние файлы. Это увеличивает размер образа и может привести к утечке данных
**Как была исправлена:** добавлен .dockerignore (теоретически)
**Как повлияло на результат:** образ меньше и безопаснее, быстрее сборка

### 5 bp

**Описание:** отсутствует USER, значит контейнер работает под root по умолчанию
**Почему плохо:** root внутри контейнера повышает риски безопасности при уязвимости приложения
**Как была исправлена:** создан отдельный пользователя и переключились на него
**Как повлияло на результат:** снижение привилегий, безопаснее эксплуатация

## Bad practices по работе с контейнерами

### 1 bad practice

**Не задавать имя контейнера (--name)**
Docker присваивает случайные имена, из-за этого неудобно управлять контейнером и искать его в списке.
Нужно задавать имя явно, например `docker run --name my-app <image>`, таким образом проще работать с контейнерами и выполнять команды.

### 2 bad practice

**Хранить данные только внутри контейнера (без volume)**

Из-за этого при удалении/пересоздании контейнера данные теряются.
Нужно использовать volume, таким образом данные сохраняются между перезапусками и пересозданием контейнеров.
`docker run -v app_data:/data <image>`
